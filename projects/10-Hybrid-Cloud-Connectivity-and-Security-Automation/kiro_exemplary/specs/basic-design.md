# 基本設計書: ハイブリッド・クラウド接続とセキュリティ自動化

## 1. システム概要

### 1.1 目的
オンプレミス環境からAWS上のデータベースに格納された機密情報へセキュアにアクセスするシステムを構築する。
VPN接続の脆弱性を悪用した不正アクセスに対し、GuardDuty + Lambda による自律防御を実現する。

### 1.2 スコープ
- オンプレミス〜AWS間のVPN接続基盤
- 機密情報DBへのセキュアアクセス経路
- 脅威検知・自動遮断の仕組み

### 1.3 前提条件
- オンプレミス環境はAWS上の別VPCで疑似的に構築する
- デプロイリージョンは `us-east-1` のみ
- 既存AWSリソースとの競合を避ける
- コストは必要最小限とする

## 3. ネットワーク基本設計

### 3.1 ゾーニング方針

- **パブリックサブネット**: インターネットアクセスが必要なリソース（AppServer、RouterPC）
- **プライベートサブネット**: 外部公開不要なリソース（RDS）
- **冗長化**: 2AZ構成（us-east-1a, us-east-1b）

### 3.2 CIDR設計方針

| VPC | CIDR | 用途 |
|-----|------|------|
| 本番VPC | 10.0.0.0/16 | AWS側本番環境 |
| 疑似オンプレVPC | 10.1.0.0/16 | オンプレ模擬環境 |

- VPC間のCIDRは重複しないこと
- /24サブネットで十分なアドレス空間を確保

### 3.3 VPN接続方針

- **方式**: AWS Site-to-Site VPN (IPsec)
- **ルーティング**: BGP（動的経路制御）
- **冗長化**: 2トンネル構成（AWS標準）
- **ルーターソフトウェア**: strongSwan (IPsec) + FRR (BGP)

## 4. セキュリティ基本設計

### 4.1 多層防御の方針

```
Layer 1: Network ACL     → サブネット境界での基本制御
Layer 2: Security Group  → インスタンス単位の最小権限制御
Layer 3: WAF + IPSet     → アプリケーション層のIP制御
Layer 4: GuardDuty       → 脅威検知 + 自動対応
```

### 4.2 アクセス制御方針

| 対象 | 方針 |
|------|------|
| Web UI | 許可IPリストからのみアクセス可能 |
| DB | AppServerからのみアクセス可能（SG制御） |
| SSH | 原則禁止（SSM Session Manager推奨） |
| VPN | IPsec暗号化 + BGP認証 |

### 4.3 自動防御の方針

- GuardDutyが脅威を検知した場合、人手を介さず自動的に攻撃元IPをブロックする
- ブロックはWAF deny-ipsetへのIP追加で実現する
- 対象脅威: 不正アクセス、偵察行為、バックドア、マルウェア等

### 4.4 IAM方針

- 最小権限の原則を厳守
- AdministratorAccess は絶対に使用しない
- Lambda実行ロールは対象リソースのARNに限定

## 5. 可用性設計

### 5.1 冗長化方針

| 対象 | 方針 | 優先度 |
|------|------|--------|
| VPNトンネル | 2本冗長（AWS標準） | must |
| サブネット | 2AZ分散 | must |
| RDS | シングルAZ（コスト優先） | may（将来Multi-AZ化） |
| EC2 | シングル構成 | may（将来Auto Scaling化） |

### 5.2 管理アクセス方針

- EC2管理はSSM Session Manager経由で行う
- アプリケーション通信を阻害しない管理経路を確保する
- SSH鍵はバックアップ手段として保持するが、通常運用では使用しない

## 6. 運用設計

### 6.1 IaC方針

- 全インフラをTerraformでコード管理する
- モジュール分割により再利用性を確保する
- 状態管理: 開発段階はローカル、本番はS3+DynamoDB

### 6.2 命名規則

- 全リソース名の先頭に `xxxx-x` プレフィックスを付与
- 形式: `xxxx-x-{用途}-{環境}-{連番}`
- タグ値は英小文字・ハイフン区切り

### 6.3 タグ方針

| タグキー | 必須 | 用途 |
|---------|------|------|
| xxxx-x | ○ | プロジェクト識別 |
| Project | ○ | プロジェクト名 |
| Environment | ○ | 環境識別 |
| Owner | ○ | 管理者 |
| ManagedBy | ○ | 管理ツール |

## 7. コスト設計

### 7.1 コスト最適化方針

| リソース | 選定 | 理由 |
|---------|------|------|
| EC2 | t2.micro | 最小構成、無料枠対象 |
| RDS | db.t4g.micro | 最小構成、Graviton2で高効率 |
| VPN | Site-to-Site VPN | 時間課金のみ |
| Lambda | 128MB / 10秒 | イベント駆動、実行時のみ課金 |

### 7.2 概算月額コスト

| リソース | 概算 |
|---------|------|
| EC2 × 3台 | ~$25 |
| RDS (db.t4g.micro) | ~$12 |
| VPN接続 | ~$36 |
| その他（EIP, データ転送等） | ~$10 |
| **合計** | **~$83/月** |
