# 学習成果報告書：モダンDevSecOpsパイプラインの構築

## 1. プロジェクト概要

外資系コンサルティングファームやメガベンチャーの技術水準をターゲットとし、単なるアプリケーション開発に留まらず、「セキュリティ」「スケーラビリティ」「可観測性（Observability）」を軸としたエンタープライズ向けのCI/CDパイプラインを構築しました。

## 2. システムアーキテクチャ

本プロジェクトでは、インフラのコード化（IaC）からコンテナオーケストレーション、継続的デリバリーまでを一貫して自動化しています。

### ディレクトリ構造

```text
11-DevSecOps-pipeLine/
├── .github/workflows/    # CI/CD (GitHub Actions) による自動化定義
├── app/                  # React フロントエンド (軽量化されたコンテナ)
├── helm/                 # Kubernetesリソース (環境差分を吸収するHelm Chart)
├── terraform/            # EKS基盤 (再利用可能なIaC)
└── monitoring/           # Prometheus/Grafana (可観測性の確保)

```

---

## 3. 実装フェーズと習得スキル

### フェーズ 1: コンテナ戦略 (React & Docker)

* **内容:** `node:alpine` を採用したマルチステージビルドの実装。
* **成果:** イメージサイズを最小限に抑え、攻撃表面（Attack Surface）を削減。軽量かつセキュアなコンテナ運用を実現。

### フェーズ 2: Infrastructure as Code (Terraform & EKS)

* **内容:** Terraform を用いた Amazon EKS クラスターの定義。
* **成果:** マネージド型ノードグループの採用、および **IRSA (IAM Roles for Service Accounts)** の設定により、Pod 単位での最小権限の原則（Least Privilege）を適用。

### フェーズ 3: DevSecOps パイプライン

* **内容:** GitHub Actions を用いたパイプラインの構築。
1. **Build:** アプリケーションの自動ビルド。
2. **Security Scan:** **Trivy** による静的解析。OSパッケージおよび依存関係の脆弱性を検知。
3. **Push:** 承認されたイメージのみを Amazon ECR へプッシュ。
4. **Deploy:** Helm を用いた EKS へのローリングアップデート。

### フェーズ 4: Kubernetes 管理の高度化 (Helm)

* **内容:** K8s マニフェストの Helm Chart 化。
* **成果:** 環境（検証・本番）ごとの変数管理を容易にし、デプロイの再現性とバージョン管理の効率を向上。

### フェーズ 5: 可観測性 (Prometheus & Grafana)

* **内容:** `kube-prometheus-stack` の導入とダッシュボード構築。
* **成果:** Pod ごとのリソース使用率を可視化。異常検知と MTTR（平均復旧時間）短縮のための基盤を構築。

## 4. 設計思想と選定理由

本プロジェクトでは、各技術の選定において明確な意図を持っています。

* **Trivy の採用 (シフトレフトの提案):**
ビルドプロセスの早期段階で脆弱性を検知することで、安全でないコードが本番環境へデプロイされるリスクを根本から防ぐ「シフトレフト」の概念を実践しました。
* **Helm による運用効率化:**
複雑な K8s マニフェストの直接管理を避け、テンプレート化することで、大規模開発におけるデプロイの標準化とミス削減を考慮しました。
* **可観測性の重視:**
インフラエンジニアとして、システムが動くだけでなく「正常に動き続けていることを証明できる」状態を作ることが重要であると考え、監視基盤を標準構成に含めました。
